# 台語羅馬拼音索引施工計畫

## 目標

- [ ] 讓新專案 `autocomplete` 支援台語羅馬字查詢（行為對齊舊站）。
- [ ] 採用「預建索引 + R2 靜態查詢」架構，避免查詢時做重運算。
- [ ] 以 Node/TS 腳本實作索引流程，避免引入 Perl 維運負擔。

## Phase 0：規格確認（先做）

- [x] 第一版同時支援 `TL`、`DT`、`POJ`，預設取 `localStorage.pinyin_t`（無值時預設 `TL`）。
- [x] 輸入判斷規則：以英文字輸入查詢（非中文字）時，判定為「羅馬字查詢」。
- [x] 建議 regex 草案（接受空白、連字號、撇號與數字聲調）：`/^(?=.*[A-Za-z])[A-Za-z0-9\\s\\-']+$/`。
- [x] fallback 規則：羅馬字查無候選詞時，不回退一般索引，維持既有流程進入 404 字圖頁。
- [x] 排序規則：不做「羅馬索引 + 一般索引」合併排序，兩種查詢路徑互斥。

> note:
> - 上述 regex 代表：必須至少包含 1 個英文字母；允許字元為英文字母、數字、空白、`-`、`'`。
> - 若要限制「數字僅作聲調」，可在 parser 階段再加規則（例如僅允許詞尾出現 `1-9`）。

## Phase 1：索引建置（Node 腳本）

- [x] 新增 `scripts/build-pinyin-lookup.mjs`。
- [x] 讀取台語詞典來源（`data/dictionary/ptck/*.txt` 或等價來源）。
- [x] 從台語欄位抽取可搜尋拼音 token（建議從 `trs` 起步）。
- [x] 做正規化（NFD、去 combining mark、大小寫與符號標準化）。
- [x] 建倒排索引：`term -> [title1, title2, ...]`。
- [x] 一次輸出三套索引到 `data/dictionary/lookup/pinyin/t/{TL|DT|POJ}/{term}.json`（路徑互不衝突）。
- [x] 新增 npm script（例：`build-pinyin-lookup`），並串進 build 流程。

> note:
> - 不建議直接複刻 Perl 檔；Node 腳本與現有 `scripts/build-search-index.mjs` 的維護模型一致。
> - token 化規則務必固定，避免每次重建索引結果抖動（影響快取與 diff）。
> - 請先確保輸出 key（term）是 URL-safe，必要時做 encode/decode 對齊。

## Phase 2：R2 上傳流程

- [x] 擴充 `commands/upload_dictionary.sh`，加入 `lookup/pinyin` 上傳。
- [x] 檢查缺檔保護（目錄不存在要明確報錯）。
- [x] 上傳摘要增加 `lookup/pinyin` 檔案數統計。

> note:
> - 索引檔通常是大量小檔，建議觀察上傳耗時與同步策略。
> - 預覽 bucket 與正式 bucket 路徑應一致，避免環境切換時 API 找不到檔案。

## Phase 3：Worker API

- [ ] 在 `worker/index.ts` 新增路由：
  - `/api/lookup/pinyin/:lang/:type/:term.json`
- [ ] 新增舊站相容路由：
  - `/lookup/trs/:term`
  - `/api/lookup/trs/:term`（與 `/lookup/trs/:term` 等效）
- [ ] 從 `DICTIONARY` R2 讀取對應 key：
  - `lookup/pinyin/{lang}/{type}/{term}.json`
- [ ] 統一回應格式（建議陣列字詞，找不到回空陣列）。
- [ ] 明確定義 `lookup/trs` 回應格式（需對齊舊站 autocomplete 使用情境）。
- [ ] 開放 CORS 給 `old.moedict.tw`（至少 `GET` 可用）以便跨網域測試舊前端。
- [ ] 設定快取 header（短 TTL + `stale-while-revalidate`）。

> note:
> - autocomplete 是高頻請求，cache 策略對成本與體感延遲影響很大。
> - API 不要回 500 當「查無資料」，應回 200 + 空陣列或 404 + 可預期結構（二擇一且全專案一致）。

## Phase 4：前端 autocomplete 串接

- [ ] 在 `src/components/searchbox.tsx` 增加「台語羅馬字輸入」判斷。
- [ ] 當 `lang=t` 且命中羅馬字規則時，呼叫新的 pinyin lookup API。
- [ ] 當 `lang=t` 且未命中羅馬字規則時，維持現有一般索引（`/api/index/t.json`）邏輯。
- [ ] 羅馬字查詢無候選詞時，不切回一般索引；提交後維持既有導頁邏輯進 404 字圖頁。
- [ ] 保留現有 UX（鍵盤上下選、Enter、mobile toggle、prefetch）。

> note:
> - `pinyin_t`（`TL`/`DT`/`POJ`）建議直接影響 API path 的 `type`，與舊站偏好一致。
> - 因查詢路徑互斥，前端應避免同一輸入同時發兩種請求，減少競態與閃爍。

## Phase 5：測試與驗收

- [ ] 單元測試：token 正規化、term 路徑轉換、查詢路徑判斷（羅馬/非羅馬）。
- [ ] 整合測試：輸入羅馬字可列出台語候選詞，點選可正確跳詞條頁。
- [ ] 整合測試：羅馬字查無候選詞時，提交後進入既有 404 字圖頁流程。
- [ ] 回歸測試：中文查詢、萬用字元查詢、跨語言前綴查詢不退化。
- [ ] 手動驗收清單：`TL` / `DT` / `POJ` 各跑至少 3 組案例。

> note:
> - 建議準備「黃金測資」清單（常見詞、同音詞、含 tone mark 詞）避免回歸時只能靠人工印象。
> - 先在 preview bucket 驗證快取與 CORS，再上正式環境。

## 風險與提醒（跨階段）

- [ ] 風險：大量小 JSON 導致 R2 請求數高。
  - note: 可評估第二階段改成 shard（例如每字首一檔）降低請求數。
- [ ] 風險：羅馬字正規化過度，造成誤召回。
  - note: 第一版先保守，必要時再加寬鬆比對模式。
- [ ] 風險：前端同時打兩條查詢造成閃爍與競態。
  - note: 以 requestId/cancel 機制保證只採用最後一次輸入結果。
